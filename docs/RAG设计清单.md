# RAG应用设计 Checklist

> 教育场景中RAG（检索增强生成）应用的完整设计与落地检查清单。

---

## 为什么教育AI特别需要RAG

教育场景对AI回答有极高要求：
- **不能胡说**（幻觉）—— 学生可能把AI答案当标准
- **要有出处** —— 教师需要验证AI结论的依据
- **范围可控** —— 超出课程范围的问题要明确指出

RAG通过"先检索、后生成"的机制，将AI的回答约束在特定知识库范围内，是教育AI的核心技术路径。

## Phase 1: 知识库构建

> **为什么这个阶段重要**
> 教育AI的知识库是"教材"而非"网页"，知识必须精确、有结构、可溯源。一个常见陷阱是直接用通用网页抓取工具处理教材PDF，导致公式丢失、章节错位、扫描版无法识别。我们曾经见过一个案例：某团队将扫描版历史教材直接入库，学生问"辛亥革命发生在哪一年"时，AI回答"根据资料，□亥革命发生在1911年"——OCR错误把"辛"识别成了方块，却进入了向量索引。

### 1.1 文档解析与结构化

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 1.1.1 | 是否定义了支持的文档类型（PDF/Word/PPT/视频转录）？ | ☐ | | 不同类型需要不同解析器 |
| 1.1.2 | 是否处理了多模态内容（图文混排、公式、表格）？ | ☐ | 教育文档常见 | 公式解析错误会直接误导学生 |
| 1.1.3 | 是否保留了文档的层次结构（章/节/小节）？ | ☐ | 影响上下文质量 | 学生需要知道答案出自哪一章 |
| 1.1.4 | 是否提取了文档的元数据（作者/版本/学科/年级）？ | ☐ | 用于过滤和权限 | 九年级学生不应看到高考内容 |
| 1.1.5 | 是否处理了扫描版PDF（OCR）？ | ☐ | 老旧教材常见 | 学校大量教材仍是扫描版 |

### 1.2 内容分块策略

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 1.2.1 | 是否根据内容类型选择分块策略（固定长度/语义/层次）？ | ☐ | | 数学题和语文课文分块逻辑不同 |
| 1.2.2 | 分块大小是否考虑了教育内容的完整性？ | ☐ | 不能切断知识点 | 切在"因为...所以..."中间会丢失因果 |
| 1.2.3 | 是否设置了合理的块间重叠（overlap）？ | ☐ | 通常10%-20% | 保证知识点不被切断 |
| 1.2.4 | 是否保留了分块的来源位置信息？ | ☐ | 用于答案引用 | "根据第三章第一节"增强可信度 |
| 1.2.5 | 特殊内容（公式/代码/图表）是否有特殊处理？ | ☐ | | LaTeX公式需要整段保留 |

### 1.3 质量管控

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 1.3.1 | 是否有内容准入标准（审核机制）？ | ☐ | 防止错误知识入库 | 一道错题入库会影响所有学生 |
| 1.3.2 | 是否有知识库更新机制（版本管理）？ | ☐ | 教材会更新 | 人教版2022vs2024内容有差异 |
| 1.3.3 | 是否有内容去重检测？ | ☐ | 避免重复检索 | 同一知识点在不同文档重复出现 |
| 1.3.4 | 是否监控知识库的覆盖度和质量指标？ | ☐ | 可回答率、准确率 | 知道哪些章节学生问不到 |
| 1.3.5 | 是否有脏数据清理机制？ | ☐ | | 页眉页脚/版权信息会污染检索 |

### 1.4 教育场景特需 🔵 NEW

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 1.4.1 | 是否建立了学科术语一致性映射表？ | ☐ | 不同教材对同一概念表述可能不同 | "函数"vs"映射"vs"关系"可能是同一概念 |
| 1.4.2 | 是否标注了知识的时效性（版本/年份）？ | ☐ | 教材版本更新时需明确标注 |  Pluto在2006年前是行星，之后不是 |
| 1.4.3 | 是否支持多语言内容处理？ | ☐ | 双语教学场景需要 | 国际学校需要中英文对照检索 |

> **术语一致性示例**：初中数学中，"函数"在有些教材叫"映射"，有些章节叫"对应关系"。如果学生用"映射"提问但知识库只索引了"函数"，检索会失败。建议维护一个同义词映射表，或在Embedding阶段做术语扩展。

> **知识时效性示例**：地理课本中，"太阳系有九大行星"在2006年前的教材中是正确的。如果知识库同时收录了新旧两版教材，AI需要根据教材版本给出对应答案，或在答案中标注"根据您使用的2020版教材..."。

## Phase 2: 检索策略

> **为什么这个阶段重要**
> 教育场景的检索不同于通用搜索——学生的问题往往是"不完全的"、"口语化的"，甚至包含学科术语的简写。检索策略不好，后面的生成环节再强也没用。一个常见错误是只用一个通用的Embedding模型，不做领域适配，导致"勾股定理"和"毕达哥拉斯定理"在向量空间中相距甚远，学生换种说法提问就检索不到答案。

### 2.1 向量化

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 2.1.1 | 是否选择了适合教育场景的Embedding模型？ | ☐ | 中文教育推荐BGE/Zhipu | 通用模型对学科术语理解不足 |
| 2.1.2 | 是否评估了向量维度和索引效率的平衡？ | ☐ | 通常768-1024维 | 维度太低丢失语义，太高浪费资源 |
| 2.1.3 | 是否支持动态更新向量（增量索引）？ | ☐ | 新知识加入时 | 学期中新增习题需要实时入库 |
| 2.1.4 | 是否有向量索引的备份和恢复机制？ | ☐ | | 索引损坏会导致服务完全不可用 |
| 2.1.5 | 是否测试了跨学科检索的效果？ | ☐ | 数学知识vs语文知识 | "方程"在数学和化学中含义不同 |

### 2.2 检索算法

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 2.2.1 | Top-K值是否根据场景调优？ | ☐ | 通常5-10 | K太小遗漏相关内容，K太大引入噪音 |
| 2.2.2 | 是否有相似度阈值过滤（低于阈值不召回）？ | ☐ | 避免无关内容 | 防止"相似"但实际无关的内容干扰 |
| 2.2.3 | 是否实现了混合检索（向量+关键词）？ | ☐ | 提高召回率 | 专有名词用关键词检索更精确 |
| 2.2.4 | 是否支持元数据过滤（学科/年级/章节）？ | ☐ | 教育场景必备 | 高三学生不应看到小学内容 |
| 2.2.5 | 检索延迟是否满足交互要求（<500ms）？ | ☐ | | 学生等待超过1秒会频繁刷新 |

### 2.3 重排序（Re-rank）

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 2.3.1 | 是否使用了重排序模型优化Top-K结果？ | ☐ | 可选，但推荐 | 粗排召回多，精排提升相关性 |
| 2.3.2 | 是否考虑了用户历史行为的个性化排序？ | ☐ | 可选 | 学生常问的内容类型可优先排列 |
| 2.3.3 | 重排序是否引入了不可控的偏见？ | ☐ | 需要检测 | 避免总是优先某本教材的内容 |

## Phase 3: 生成与交互

> **为什么这个阶段重要**
> 这是学生直接感知的环节，也是问题暴露最多的地方。即使检索到了正确的内容，如果生成环节失控，AI仍可能"一本正经地胡说"。一个常见错误是直接把检索到的内容塞进Prompt，不做任何格式要求和范围约束，导致AI自由发挥、添加不存在的信息。我们曾见过一个案例：检索到"光合作用需要阳光"的教材内容，AI却在答案末尾加了"所以晚上植物会死"——这既不是检索内容，也不符合科学事实。

### 3.1 上下文组装

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 3.1.1 | Prompt是否包含明确的角色设定？ | ☐ | "你是课程助手，只基于提供的资料回答" | 明确角色减少自由发挥 |
| 3.1.2 | 是否限制了回答范围（超出知识库不回答）？ | ☐ | 核心安全机制 | 防止AI编造超出课程范围的内容 |
| 3.1.3 | 是否包含了格式要求（结构化输出）？ | ☐ | 要点/步骤/总结 | 结构化答案更易阅读和理解 |
| 3.1.4 | 上下文窗口是否合理利用（不溢出）？ | ☐ | 注意token限制 | 超出token限制会丢失检索内容 |
| 3.1.5 | 是否保留了引用标记用于答案溯源？ | ☐ | | 学生需要知道答案来自哪一页 |

### 3.2 答案生成

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 3.2.1 | 是否设置了温度参数（低温度减少幻觉）？ | ☐ | 教育场景推荐0.1-0.3 | 低温减少创造性幻觉 |
| 3.2.2 | 是否限制最大生成长度？ | ☐ | 防止冗长 | 学生注意力有限，需要简洁 |
| 3.2.3 | 是否有后处理（格式校验/敏感词过滤）？ | ☐ | | 过滤不当内容和格式错误 |
| 3.2.4 | 是否支持多轮对话的上下文管理？ | ☐ | | 学生追问需要理解前文 |
| 3.2.5 | 不同学科是否需要不同的prompt模板？ | ☐ | 理科vs文科 | 数学需要步骤，语文需要赏析 |

### 3.3 可信度设计（最关键）

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 3.3.1 | 答案是否附带来源引用（文档+页码）？ | ☐ | 教育场景必须 | 教师需要验证答案依据 |
| 3.3.2 | 是否显示置信度或可靠性标记？ | ☐ | 高/中/低 | 学生知道哪些答案需要谨慎对待 |
| 3.3.3 | 超出知识库范围的问题是否有明确提示？ | ☐ | "这不在我的知识范围内" | 防止AI硬编答案 |
| 3.3.4 | 不确定的答案是否有降级表达？ | ☐ | "根据资料，可能..." | 降低学生全盘接受的风险 |
| 3.3.5 | 是否有用户反馈机制（答案好不好）？ | ☐ | 用于持续优化 | 学生点踩是最直接的信号 |

### 3.4 兜底机制

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 3.4.1 | 检索失败时（无相关文档）是否有兜底回复？ | ☐ | | 空检索不能返回空页面 |
| 3.4.2 | 生成异常时（重复/乱码/敏感）是否有拦截？ | ☐ | | 保护学生不受异常内容影响 |
| 3.4.3 | 是否有人工客服/教师介入的转接机制？ | ☐ | 关键问题人工兜底 | 复杂问题需要真人判断 |
| 3.4.4 | 是否记录了所有异常case用于后续优化？ | ☐ | | 异常case是最宝贵的优化线索 |

### 3.5 教育场景特需 🔵 NEW

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 3.5.1 | 数学公式是否正确渲染（LaTeX/MathML）？ | ☐ | 需要验证前端渲染效果 | 学生看不清公式就无法验证 |
| 3.5.2 | 是否支持追问和渐进式解释？ | ☐ | 学生可能需要"再简单一点" | 一刀切的答案不适合所有学生 |
| 3.5.3 | 是否有"鼓励性语言"设计？ | ☐ | 教育场景需要正向反馈 | "你的思路很棒"比"正确"更有激励作用 |

> **渐进式解释示例**：学生问"为什么三角形的内角和是180度？"第一次回答可以给出证明过程；如果学生追问"还是没懂"，则切换到折叠纸片的直观演示；再追问可以用动画辅助。每一层都应该检测学生的理解程度，而不是一次性倾倒所有信息。

> **鼓励性语言示例**：当学生回答错误时，不说"错了"，而说"这个思路很有趣，再想想如果...会怎样？"；当学生回答正确时，不说简单的"对"，而说"完全正确！你抓住了这个定理的核心！"正向反馈能显著提高学生的持续使用意愿（经验值：正向反馈使学生留存率提升约25%）。

## Phase 4: 评估与迭代

> **为什么这个阶段重要**
> 教育AI上线不是终点，而是持续优化的起点。学生的提问模式、教材的更新、教学大纲的调整都会影响RAG的效果。一个常见错误是上线后不做持续监控，直到有教师投诉"AI教错了"才发现问题——此时可能已经影响了数百名学生。建议至少每月进行一次抽样评估，每季度进行一次全面评估。

### 4.1 效果评估

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 4.1.1 | 是否有可回答率指标（用户问题有多少能回答）？ | ☐ | 目标>85% | 太低说明知识库覆盖不足 |
| 4.1.2 | 是否有答案准确率评估（人工标注）？ | ☐ | 定期抽样 | 自动评估不够，需要人判断对错 |
| 4.1.3 | 是否有幻觉率监控？ | ☐ | 目标<5% | 教育场景幻觉容忍度极低 |
| 4.1.4 | 是否有用户满意度反馈？ | ☐ | | 学生觉得好用才是真的好 |
| 4.1.5 | 是否有与上一版本的对比评估？ | ☐ | 防止退化 | 新版本不能比旧版本差 |

### 4.2 持续优化

| # | 检查项 | 状态 | 备注 | 为什么 |
|---|--------|------|------|--------|
| 4.2.1 | 是否有知识库定期更新机制？ | ☐ | | 教材更新后知识库必须同步 |
| 4.2.2 | 是否根据用户反馈优化分块策略？ | ☐ | | 用户问不到的内容需要调整分块 |
| 4.2.3 | 是否定期更新Embedding模型？ | ☐ | | 新模型可能大幅提升检索效果 |
| 4.2.4 | 是否有A/B测试框架验证优化效果？ | ☐ | | 凭感觉优化容易适得其反 |

## 最小可行RAG（MVP版）

> 适用场景：1个工程师，2周时间，需要快速验证教育RAG的可行性。
> 核心思路：**够用 → 好用 → 完美**，先让第一个版本跑起来。

### MVP目标定义

| 维度 | MVP目标 | 完美版目标 |
|------|---------|-----------|
| 可回答率 | >70% | >85% |
| 幻觉率 | <10% | <5% |
| 响应延迟 | <3秒 | <500ms |
| 来源引用 | 有文档名 | 有文档名+页码 |
| 多轮对话 | 不支持 | 支持 |
| 人工兜底 | 教师微信群反馈 | 系统化转接机制 |

### MVP技术选型建议

| 组件 | 推荐方案 | 替代方案 | 预估搭建时间 |
|------|---------|---------|-------------|
| **文档解析** | Unstructured / Marker | LlamaParse | 半天 |
| **Embedding** | BGE-M3（开源，中文教育友好） | Zhipu Embedding API | 1小时 |
| **向量数据库** | Chroma（本地零配置） | Milvus / Qdrant | 2小时 |
| **LLM** | GPT-3.5 / GLM-4（API调用） | 本地部署Qwen2.5-7B | 半天 |
| **框架** | LangChain（文档丰富，社区大） | LlamaIndex | 半天 |
| **部署** | Streamlit / Gradio（快速Demo） | FastAPI + 前端 | 1天 |

### MVP两周排期

**Week 1：让基础流程跑通**

| 天数 | 任务 | 交付物 |
|------|------|--------|
| Day 1-2 | 选1本教材（建议选结构清晰的PDF），完成解析和分块 | 能打印出分块后的文本 |
| Day 3 | 接入Embedding模型，生成向量，存入Chroma | 能用自然语言查询到相关段落 |
| Day 4-5 | 接入LLM，写基础Prompt（角色+范围限制+引用要求） | 能端到端回答问题并标注来源 |

**Week 2：让MVP可用**

| 天数 | 任务 | 交付物 |
|------|------|--------|
| Day 6-7 | 加兜底机制（检索为空时提示/生成异常时拦截） | 异常输入不会崩溃 |
| Day 8-9 | 做简单评估：准备20道测试题，人工判断答案质量 | 评估报告（可回答率+准确率） |
| Day 10 | 部署Demo，找3-5个真实学生试用，收集反馈 | 用户反馈记录 |

### MVP只做这些（P0检查项）

```
Phase 1: 知识库构建
  □ 1.1.2 处理多模态内容（至少支持文本+公式）
  □ 1.1.4 提取文档元数据（学科/年级）
  □ 1.2.2 分块不切断知识点
  □ 1.2.4 保留分块来源位置
  □ 1.3.1 内容准入审核（人工过一遍）
  □ 1.4.1 术语一致性（至少维护一个同义词表）

Phase 2: 检索策略
  □ 2.1.1 选择教育友好的Embedding模型
  □ 2.2.2 相似度阈值过滤
  □ 2.2.3 混合检索（向量+关键词）
  □ 2.2.4 元数据过滤（至少支持学科过滤）

Phase 3: 生成与交互
  □ 3.1.1 Prompt角色设定
  □ 3.1.2 限制回答范围
  □ 3.3.1 答案附带来源引用
  □ 3.3.3 超出知识库范围明确提示
  □ 3.4.1 检索失败兜底回复
  □ 3.4.2 生成异常拦截
  □ 3.5.1 数学公式正确渲染

Phase 4: 评估与迭代
  □ 4.1.1 可回答率评估（20道题测试）
  □ 4.1.3 幻觉率人工判断
```

### 从MVP到完美版的路径

```
MVP（Week 2）
  ↓
好用（Month 1-2）
  ├─ 支持多本教材
  ├─ 多轮对话
  ├─ 用户反馈机制（👍/👎）
  ├─ 响应速度优化到<1s
  └─ 术语一致性自动检测
  ↓
完美（Month 3-6）
  ├─ 知识库自动更新
  ├─ 个性化推荐
  ├─ A/B测试框架
  ├─ 人工介入系统
  └─ 全学科覆盖
```

### MVP避坑指南

| 坑 | 解决方案 |
|----|---------|
| 想一步到位做完美分块 | 先用固定长度512token，后续再优化 |
| 纠结Embedding模型选型 | BGE-M3直接上，后面可以换 |
| Prompt写了100行 | 先写10行核心约束，迭代添加 |
| 试图覆盖所有教材 | 先搞定1本，验证可行性再扩展 |
| 忽视学生真实反馈 | Week 2就找真实学生试用，不要只自己测 |
